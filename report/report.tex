\section{Data Collection / Manipulation}

Generative AI struggles with understanding and parsing data from the Rail Data Marketplace. This is likely because the data is provided by real systems that have complicated documentation that is not included in the models training set giving it no reference points of how to understand the formats presented. The AI is dumb.

\noindent
\newline
Our data was collected from a number of data sources provided by NRE (National Rail) and NWR (Network Rail) through the RDM (Rail Data Marketplace). The following resources were heavily used.

\begin{itemize}
  \item Network Rail Corpus
  \item Network Rail Schedule
  \item National Rail Knowledge Base Stations
  \item National Rail Darwin Push Port (Archive Provided By David Wheatley)
  \item National Rail Darwin Timetable
\end{itemize}

\noindent
Once our data was collected in its respective formats we could use python scripts provided in the \texttt{collection} folder to generate the SQL database needed for our API.

\noindent
\newline
The first operation we must undertake is the creation of the \texttt{stations} table. We can run the Python module with the following command.

\noindent
\newline
\texttt{python -m collection.stations.generate}

\noindent
\newline
This command will generate the stations table combining data from Network Rail Corpus and the National Rail Knowledge Base Stations.

\section{Architecture}

For my API I chose to use the Go programming language. This choice was made on the basis that Go is a compiled static typed language which allows it to be highly performant when compared to APIs defined in other interpreted programming languages such as Python or JavaScript.

\noindent
\newline
To make development of my API in Go simpler I chose to employ an API framework. In my case I chose to use a common well documented framework called Gin. This choice meant there was plenty of documentation of how to use the framework and that a LLM would likely have a good amount of training data related to the framework to infer from.

\noindent
\newline
For my database I chose to use PostgreSQL, this choice was made after trial testing with SQLite. SQLite started to show extreme slowdown during the generation of the dataset, mainly the \texttt{stops} table, this forced me to reconsider my choice of DBMS. Instead of operating as a single portable file which is programmatically controlled through a driver PostgreSQL provides a full fledged engine making it much more performant and able to handle large volumes of traffic. Along with this PostgreSQL allows for more fine grained control of data types such as the use of \texttt{VARCHAR}, reducing the overall storage usage of the database.

\section{Authentication}

Endpoints in my API are organised into three different levels of authorisation.

\begin{itemize}
  \item Product - This group covers endpoints that relate to the actual core date API provided, authentication at these endpoints does not require login.
  \item Administration - Covers endpoints used by administrators to perform actions such as updating an application approval status.
  \item Protected - Covers endpoints that the customer should manage, i.e. creating new applications, refreshing application keys.
\end{itemize}

\noindent
Product endpoints rely on the Bearer authorisation scheme, this means that an application key can be used by an autonomous user to make requests. This allows for the development of external applications which make use of our API.

\noindent
\newline
Administration endpoints make use of JWT (JSON Web Token) authorisation. A given administrator can login and retrieve a JSON Web Token. This token can then be used on subsequent requests to administration endpoints. In the JWT claims the \texttt{"type"} attribute is set to \texttt{administrator}, this allows us to check that the given requester should have access to the administrative endpoints.

\noindent
\newline
Protected endpoints make use of the same JWT (JSON Web Token) authorisation. Once we have verified the validity of the token we can check that the customer requesting data has authorisation to access the customer data they are requesting. This is usually that the requested data is the customers own.

//https://datatracker.ietf.org/doc/html/rfc6750

\noindent
\newline
To sign JWTs (JSON Web Tokens) I have chosen to use a public private Ed25519 key pair and the EdDSA digital signature scheme. This choice was made as Ed25519 presents one of the most secure signature schemes currently adopted in wide use while still being performant for low tier hardware.

\section{Generative AI Declaration}

Generative AI was used to aid with the development of both the API application itself and the data manipulation scripts. The following Generative AI tools were used.

\begin{itemize}
  \item T3 Chat (\href{https://t3.chat}{t3.chat})
\end{itemize}

\noindent
The T3 Chat tool allows usage of a wide range of LLM models in a chat interface which aided with the development of the application through prompts phrased as questions.

\noindent
\newline
The primary model used through this tool was Claude Sonnet 4.6 due to its high LiveBench (\href{https://livebench.ai}{livebench.ai}) score and balanced token cost.
